# KEEL Glossary

**KEEL** — Knowledge-Encoded Engineering Lifecycle. A structured process for agent-driven software development where humans steer and agents execute through specialized pipelines. Adapted from OpenAI's harness engineering.

**Knowledge-Encoded** — The "K" in KEEL. The principle that everything agents need must be committed as versioned artifacts in the repository. If it's not in the repo, it doesn't exist.

**Knowledge Boundary** — The limit of what an agent can see. Anything outside the repo (Slack, Google Docs, verbal decisions, tacit expertise) is invisible. The only way to make it visible: encode as markdown in the repo.

**Agent Legibility** — Optimizing documentation and code for agent comprehension rather than human aesthetics. Specs must be unambiguous enough for agents to execute without clarification.

**Progressive Disclosure** — Information architecture where agents start with a small, stable entry point (the project guide ~80 lines) and are taught where to look next, rather than being overwhelmed with everything up front.

**North Star** — The vision document (`NORTH-STAR.md`, at the project root) that defines where the project is heading, what principles govern decisions, and how the process evolves through growth stages. Where taste is encoded before it becomes linters.

**Handoff** — A per-feature directory (`docs/exec-plans/active/handoffs/WI##-<slug>/`) that persists context between pipeline agents. It holds `routing.json` (orchestrator-owned pipeline state, mutated only via `scripts/keel-routing.py`), `resolved-work-item.json` (the resolver's immutable Step 0 output), one `<agent>.md` per contributing agent (each agent self-writes its own with the `Write` tool), and `<touchpoint>-review/` deliberation subdirs. Agents `Read` the sibling files they need; there is no single shared file. Agent files are SNAPSHOT (overwritten whole on re-run); deliberation subdirs are APPEND-only (`attempt-NN.md` per pass). Revision history lives in deliberation attempts plus `git log`. See `docs/process/HANDOFF-CONTRACT.md`.

**Pipeline Variant** — One of four execution paths a feature takes through the agent roster:
- **Bootstrap:** One feature per bootstrap entry — those tagged `Binder-exempt: bootstrap`, a variable set (one scaffold + test-infra for a single-package stack, one scaffold per package plus test-infra for full-stack), not a fixed WI01–WI03 triple. Each dispatches the agent named in its entry then landing-verifier then review? then post-landing:
  - scaffolder → landing-verifier → review? → post-landing (one per package)
  - config-writer → landing-verifier → review? → post-landing (test infra)
- **Backend:** pre-check → review-precheck? → researcher? → arch-advisor-consult? → backend-designer? → review? → test-writer → implementer → [code-reviewer ∥ spec-reviewer ∥ safety-auditor? ∥ arch-advisor-verify?] → landing-verifier → review? → post-landing
- **Frontend:** pre-check → review-precheck? → researcher? → arch-advisor-consult? → frontend-designer → review? → test-writer → implementer → [code-reviewer ∥ spec-reviewer ∥ arch-advisor-verify?] → landing-verifier → review? → post-landing
- **Cross-cutting:** pre-check → review-precheck? → test-writer → implementer → code-reviewer → landing-verifier → review? → post-landing

(Maintenance work is **not** a variant — it lands through the Maintenance lane, below.)

**Maintenance lane** — The sanctioned path for repo changes that are not features: dependency/lockfile bumps, tooling-generated config, formatter runs, ignore/editor-config edits, license headers, typo sweeps. A *sibling to* the four pipeline variants, not a fifth — it runs no agent handoff, so it writes nothing to `routing.json`; the typed `chore:`/`infra:` commit is its record. Flow: adopt-or-make the change → green gate (tests + lint) → doc-gardener ad-hoc → commit. No Binder, resolved-work-item file, or required backlog entry. Qualification (the admissibility boundary that keeps it from laundering a product feature), procedure, and the `Maintenance review` knob are defined in `docs/process/PIPELINE-DOCTRINE.md` §"The maintenance lane". Lands `Binder-exempt: infra` and `Binder-exempt: trivial` cards.

**Karta lane** — The sanctioned dialed-down-rigor path for *building a feature fast while keeping a visible ledger of every cut*. A permanent *sibling to* the full-rigor `keel-*` lane (not a fifth pipeline variant, not a bypass, and not a one-way graduation): `/karta-refine` → `/karta-pipeline WI##`, one feature per run, on the same Binder shape. It drops the design agents, code-reviewer, the keel spec-reviewer (replaced by its delta `karta-spec-reviewer`), arch-advisor, `landing-verifier`, and the review panel, and is **spec-first** (tests secondary, declared with a marker where skipped) — but keeps a **non-negotiable floor**: the `karta-spec-reviewer` structural conformance gate, the green gate (the project's configured tests/build, run as a direct orchestrator command sourced from the project guide — RED, or no runnable command, halts fail-closed), `safety-auditor`, the two write-time sign-off halts (`KARTA-PLACEHOLDER` / `KARTA-GUARD`), and bones-clean admissibility. Lean map: pre-check → implementer → karta-spec-reviewer → safety-auditor → green gate → complete. Every deferred cut is declared inline with a declared-debt marker (below). You choose per feature by which command you run. Full discipline: `docs/process/KARTA-LANE.md`.

**Declared-debt markers (`KARTA-DEFER` / `KARTA-PLACEHOLDER` / `KARTA-GUARD`)** — The three inline code markers the Karta lane uses to record what rigor was cut, at the cut site. `KARTA-DEFER` is a deferred shortcut (skipped test-first, hardcode, punted edge) and does not halt; `KARTA-PLACEHOLDER` is a mock/stub/fake and halts for a human OK before it is written; `KARTA-GUARD` is a destructive, irreversible, or PII/secret-touching effect and halts regardless of whose infra hosts it. They record declared *debt*, not lane provenance — a feature that deferred nothing leaves no marker. Grammar (`TOKEN: <what> | real: <intended> | finish: <hint>`), the integrity rules, and the `scripts/karta-deferred-ledger.py` scanner are in `docs/process/PIPELINE-DOCTRINE.md` §"Declared-debt markers".

**Binder-exempt: infra** — A backlog card for build, CI, tooling, or dependency work with no product behavior of its own. Lands via the Maintenance lane.

**Binder-exempt: trivial** — A backlog card for a behavior-preserving change too small to spec — a typo, a formatter run, a config value with no runtime effect. Lands via the Maintenance lane. (Distinct from the `trivial` *complexity tier*, which is a pre-check routing decision for a real feature.)

**Execution Brief** — The structured output of the pre-check agent. Contains: intent classification, complexity tier, spec reference, dependencies, what to build, new/modified files, acceptance tests, edge cases, risks, constraints for downstream (MUST/MUST NOT), and routing decisions (designer needed? researcher needed? arch-advisor needed?).

**Orchestrator** — The human who steers the KEEL process: kicks off features and reviews the completed work. The `keel-pipeline` skill runs the pipeline end-to-end: dispatching agents, running the review panel (personas by default, roundtable opt-in), running the post-landing procedure (doc-gardener, archive the handoff to `completed/handoffs/`, commit on the feature branch). Completion is repo-local — code committed on the `keel/WI##-<slug>` branch, handoff archived, backlog `[x]`; no forge is required. Pushing the branch and opening a PR are separate ceremony, run by the human via `/keel-submit` when their workflow uses a forge. The orchestrator does not write code.

**Invariant** — A non-negotiable rule specific to the project's domain, enforced mechanically. Examples: "never force-pull" (git), "validate all input at boundaries" (API), "all transforms must be idempotent" (data pipeline).

**Golden Principle** — An opinionated, mechanical rule that keeps the codebase legible and consistent for future agent runs. Encoded in the repo and enforced continuously. From OpenAI: "Human taste is captured once, then enforced continuously on every line."

**Garbage Collection** — Periodic sweeps to detect and fix documentation drift. The doc-gardener agent scans for stale content; the orchestrator applies fixes. "Docs that lie are worse than no docs."

**Ralph Wiggum Loop** — Agent-to-agent review pattern where an agent reviews its own changes, requests additional agent reviews, and iterates until all reviewers are satisfied. Named by OpenAI. Enables increasing autonomy without human bottleneck.

**Oracle** — The per-feature block in the JSON Binder (`work_items[].oracle`) that names what makes the feature pass or fail. Required keys: `type` (`unit | integration | e2e | smoke`) and `assertions[]`; optional `tooling`, `setup`, `actions[]`, `gating`. Consumed by test-writer (translates `assertions[]` into failing tests), pre-check (emits each assertion as a JSON-pointer `/work_items/<idx>/oracle/assertions/<aidx>` in the execution brief), spec-reviewer (verifies implementation conforms), safety-auditor (scans the oracle for auth/credential/token surface that must be checked against domain invariants), and backlog-drafter (synthesizes one for each new feature). Paired with `contract` on the same feature: `contract` declares the surface (what exists), `oracle` declares what must hold true about it. The name is the formal-methods term-of-art (Howden, Weyuker) for the authoritative source of correctness, distinct from the tests that exercise it.

**RED → GREEN Flow** — The handoff between test-writer and implementer. Test-writer produces failing tests (RED state). Implementer writes code to pass them (GREEN state). Neither crosses the boundary: test-writer never writes implementation, implementer never modifies tests.

**Lifecycle** — The "L" in KEEL. The full arc: north star → spec → backlog → pipeline → landed feature → garbage collection. Every feature goes through this complete cycle.

**Arch-advisor** — Read-only architecture consultant agent with two invocation modes: CONSULT (Step 1.7, before design) provides architecture guidance for complex features; VERIFY (in Step 5's parallel review gates, before landing) performs independent structural review. Gated by pre-check's complexity classification — only runs for architecture-tier features.

**Intent Classification** — Pre-check's mandatory first step. Categorizes work as refactoring, build, mid-sized, architecture, or research. Determines pipeline routing: which optional agents run, complexity tier, and whether Arch-advisor is needed.

**Complexity Tier** — Pre-check's assessment of feature scope: trivial (skip designer), standard (normal pipeline), complex (all gates), architecture-tier (Arch-advisor consultation + verification). Drives pipeline routing decisions.

**Wisdom Accumulation** — Pattern where agents propagate context downstream through structured Decisions (choices made and why) and Constraints (MUST/MUST NOT for downstream agents) in their per-agent files within the feature's handoff directory. Prevents agents from repeating upstream mistakes or violating upstream decisions.

**Structured Rejection** — Pattern where gate agents (spec-reviewer, safety-auditor) output a machine-readable `**Verdict:**` field as their first line. The pipeline branches on this verdict. Max 2 spec-review loops, max 3 safety-auditor loops before escalating to human.

**VERIFIED** — The landing-verifier gate verdict: all pipeline gates passed and tests pass, but the feature has not yet been committed. The orchestrator runs the landing review (Step 8.5, always — the review panel is the default) and then the post-landing procedure (Step 9) to complete the feature repo-locally: commit on the feature branch and archive the handoff to `completed/handoffs/`. (`VERIFIED` is a gate verdict, not a routing status — the handoff status enum is `IN-PROGRESS | BLOCKED`.)

**Pragmatic Minimalism** — Arch-advisor's core decision framework: bias toward simplicity, leverage what exists, prioritize developer experience, one clear path, match depth to complexity. Ported from OMA (Oh My OpenAgent).

**Two-Org Test** — Proposal-evaluation gate from `AGENTS.md` §"Framework proposal design: defaults + knobs". A KEEL framework knob is licensed only when the proposer can name two real organizations that would make different durable choices for non-pathological reasons. Bikeshedding ("someone could prefer X") doesn't earn a knob. Distinguishes documented disagreement-in-practice from hypothetical preference.

**Least-Assuming Default** — From `AGENTS.md` §"Framework proposal design: defaults + knobs". The behavior requiring the fewest org-specific assumptions, not the cleanest UX or the proposer's preferred behavior. When a knob's default would force an org-specific assumption (rebase vs merge vs squash, monorepo vs polyrepo, gh vs forge-CLI), the default must be the path that assumes least; the assumption-laden behavior is opt-in.

**Review Panel** — The umbrella term for the adversarial, multi-perspective review the pipeline applies at three checkpoints: after pre-check (Step 1.3), after design (Step 2.5), and after landing (Step 8.5). Exactly one of two interchangeable panels serves all three of a feature's checkpoints — resolved once at Step 0.5 into `routing.json`'s `review_panel` — and both emit the same `APPROVED | CONCERNS` verdict and kickback semantics, so pipeline steps stay panel-agnostic. The persona panel is the default: in-process review by four parallel `review-panelist` sub-agents (one per review lens), needing nothing but the host. Roundtable (see below) is the opt-in external alternative. Selection is governed by `Review panel: personas|roundtable` in the project guide (absent ⇒ `personas`), overridable per feature by a `Review:` field on the backlog entry. The panel is advisory: a `CONCERNS` verdict kicks back to the artifact's authoritative author (pre-check for routing, the designer for the blueprint, the landed code for landing) for revision rather than overwriting it or halting on its own. Review always runs and is never silently skipped. This is the review the Orchestrator and VERIFIED entries refer to. See `docs/process/REVIEW-PANEL.md`.

**Roundtable Review** — The opt-in external arm of the Review Panel: multi-model review over the `roundtable` MCP server, adding cross-vendor diversity at the cost of external round-trips. Selected per project or per feature via the Review Panel knob (see above). When selected, it runs at the same three pipeline touchpoints — post-pre-check (Step 1.3, tools `roundtable-critique` + `roundtable-canvass`), post-designer (Step 2.5, `roundtable-blueprint` + `roundtable-critique`), and pre-landing (Step 8.5, `roundtable-crosscheck` + `roundtable-critique`) — plus four encoded-draft gates that have no persona equivalent: `/keel-setup` (architecture and invariants), `/keel-adopt` (invariants), and `/keel-refine` (Phase 3.5 backlog decomposition). At any of these seven sites the orchestrator may invoke `roundtable-converge` as an opt-in reactive second round when the panel splits — a peer-redacted replay in which each panelist re-evaluates with the others' reasoning visible and returns a hold-or-revise stance, an agreement list, and a draft converged recommendation. Roundtable is advisory, not authoritative — findings feed back through pre-check, the designer, or the existing gate chain and never directly block landing. It is never silently skipped: if the MCP server is unavailable, review falls back to the persona panel. See `docs/process/REVIEW-PANEL.md`.

**Work Item (WI##)** — The smallest independently-testable, vertical-slice node (id `WI##`); a bug, task, or feature are all Work Items. It is the slice node in the backlog dependency DAG and the pipeline's unit of execution (one Work Item flows through one `/keel-pipeline` run); cross-slice cohesion comes from the shared `Binder: <slug>`. **A KEEL Work Item is not a Scrum "user story" and need not be user-facing** — the test that selects the unit is *testability in isolation*, not the presence of an end user; agents must not force "As a user…" framing onto work that has no user. See `backlog.md` and Step 0 of the keel-pipeline skill.

**Binder (`<slug>.json`)** — A bounded body of related work that decomposes into Work Items: the structured JSON planning artifact (`docs/exec-plans/binders/<slug>.json`, validated against `schemas/binder.schema.json`) holding shared scope, design facts, invariants, and the Work Item array (`work_items[]`) with their contracts and oracles. Authored by `/keel-refine`; the pipeline reads exactly this one input shape. Backlog Work Items link to their Binder via `Binder: <slug>` (or declare `Binder-exempt: <reason>`). Named so it maps onto common planning tools rather than for any universal vocabulary; see **Interop** below for the exact adapter table.

**Initiative** — A purely *external* mapping target: the top tier of common planning taxonomies (a collection of Binders pursuing one larger outcome). KEEL names the tier only so a client's Initiative maps onto KEEL — it owns no Initiative file, schema, backlog field, or pipeline behavior in the repo (P3: current state is self-sufficient, and there is no Initiative state to reconstruct). When a client's Initiative yields a **requirements brief** — its raw input to `/keel-refine` (prose, markdown, a wireframe bundle, or pasted images) — `/keel-refine` consumes it and its KEEL-owned *output* is the Binder. The canonical chain: Initiative (client-owned) → requirements brief (raw input) → `/keel-refine` → Binder (`<slug>.json`) → Work Items (`WI##`); the requirements brief is input only, never a repo-owned artifact or field. A client that organizes work this way owns the grouping in their own tracker; KEEL's only repo-owned tiers are Binder (`<slug>.json`) and Work Item (`WI##`). See **Interop** below.

**Interop** — KEEL's taxonomy is named to *map onto* widely understood planning vocabularies, not to be a universal standard. Only Binder and Work Item are repo-owned; Initiative is mapping-only (no KEEL state). The adapter table:

|KEEL tier|Jira|Linear|GitHub|Repo-owned by KEEL?|
|-|-|-|-|-|
|Initiative|Initiative|Initiative|—|No — external mapping target only|
|Binder|Epic|Project|Milestone|Yes — `docs/exec-plans/binders/<slug>.json`|
|Work Item|Story / Task / Bug|Issue|Issue|Yes — `WI##` in `backlog.md`|

Linear has no "Epic" tier; KEEL's Binder maps to Linear's **Project** and GitHub's **Milestone**. A KEEL Work Item maps to whichever Jira issue type fits the work (Story, Task, or Bug) and to a Linear/GitHub **Issue** — KEEL does not distinguish them (see **Work Item**).

**Branching Policy** — the project guide "Pipeline Preferences" key controlling how /keel-pipeline WI## handles unmerged `Needs:`. `halt` (default — least-assuming) refuses to start WI## with unmerged Needs and emits a CTA. `stack` (opt-in) branches WI## from the unmerged intra-Binder ancestor's tip, decoupled from any remote — the stack is branch-based (`parent_branch`/`parent_sha`). Cross-Binder Needs always halt regardless of policy. See Step 0 of the keel-pipeline skill.

**Restack** — The `git rebase --update-refs --onto <base> <parent_sha>` operation that lifts a stacked feature branch onto base after its parent merged. Universal across squash/merge/rebase merge strategies; requires Git ≥ 2.38. /keel-pipeline runs the restack at re-invocation when it detects `parent_sha` has become an ancestor of base.
