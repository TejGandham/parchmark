**Concrete snippet for `49f4bec52ca3.upgrade()` (from arch-advisor consultation §A):**

```python
def upgrade() -> None:
    """Convert embedding column from JSON to pgvector Vector(1536).

    Brownfield-tolerant per backend/migrations/versions/3c1162fce719_add_oidc_support_to_user_model.py
    pattern. Fresh DB: no-op (Base.metadata.create_all already produced post-7f1c schema).
    Brownfield at parent revision 33a4da6b0cac: convert JSON → vector(1536) via raw DDL.
    """
    conn = op.get_bind()
    inspector = sa.inspect(conn)
    if not _table_exists(inspector, "notes"):
        return  # fresh DB; create_all has handled schema
    existing = _get_existing_columns(inspector, "notes")
    if "embedding" not in existing:
        return  # past this revision in some other shape
    # Brownfield with JSON-typed embedding — perform conversion via raw DDL.
    op.execute("CREATE EXTENSION IF NOT EXISTS vector")
    op.execute("ALTER TABLE notes ADD COLUMN embedding_vec vector(1536)")
    op.execute(
        "UPDATE notes "
        "SET embedding_vec = embedding::text::vector(1536) "
        "WHERE embedding IS NOT NULL"
    )
    op.drop_column("notes", "embedding")
    op.alter_column("notes", "embedding_vec", new_column_name="embedding")
```

Mirror the same shape for `7f1c343772e8.upgrade()` and `.downgrade()` — the only material difference is which columns/extension are added vs dropped.

**Helper functions — implementer's choice:**

The brownfield guards need `_table_exists(inspector, table)` and (for `49f4`) `_get_existing_columns(inspector, table)`. Two acceptable approaches:

- **Inline duplication**: define both helpers at module scope in each of the two patched migrations (matches `3c1162fce719:27-36` and `33a4da6b0cac:21-28` style — duplication is already the norm). Lowest blast radius, easiest to review.
- **Shared module**: extract to `backend/migrations/helpers.py` and import from both files. Cleaner long-term but introduces a new shared-import edge into the migrations dir, which historically has avoided shared module dependencies. If the implementer takes this route, they MUST ensure the helpers module is importable from within `env.py`'s `sys.path` configuration and that no migration imports it via a `from .helpers import _table_exists` relative-import (alembic loads revision files via a custom loader that does not always honor package context).

Both are acceptable. Pre-check does not prescribe which.

**CLAUDE.md addition:**

Append a new section titled "Migration history conventions" near the existing "## Gotchas → Migrations" subsection (or as a sibling top-level section if that fits the file's structure better). Content (see contract.claude_md_addition for the canonical phrasing; implementer should match wording closely):

- **Immutable across the project lifetime**: `revision`, `down_revision`, `branch_labels`, `depends_on` identifiers; the file's revision filename; the *intent* and *direction* of each migration.
- **Patchable only for dep-removal hygiene**: module-level imports and function bodies — provided the generated SQL on a *brownfield* database remains semantically equivalent (same columns added/dropped, same extension state). The change must be reviewable as "removing a load-time dependency on a now-absent package," not as "fixing a logic bug" or "cosmetic cleanup."
- **Required when patching**: the migration must be tested against (a) a fresh DB (returns no-op early) and (b) a brownfield DB at the parent revision (produces the same end state as before the rewrite).
- **Explicit non-goal**: not a green light to retroactively edit migrations for cosmetic refactors, ruff cleanups, type-hint modernisation, comment rewrites, or general tidiness. Dep-removal hygiene is the entire scope.
